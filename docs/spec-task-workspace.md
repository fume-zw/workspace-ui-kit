## 1. 目的

採用管理サンプルを土台に、**プロジェクト・タスク業務**で使える **4 ペインワークスペース**を構築する。既存の Supabase タスク管理ツールと連携し、**スマホからの軽いタスク追加**と **PC での深い作業**を両立する。

## 2. 背景と前提

### 2.1 本リポジトリ（workspace-ui-kit）

- Next.js 16 / React 19 / shadcn/ui（base-nova）の **4 ペイン雛形**。
- 現状のデータは `data/*.json` を Server Component で読み込む構成（`app/page.tsx`）。
- 教材上の方針として、**DB 接続・認証は次フェーズ**とされているが、本仕様では Supabase 連携を前提に設計する。

### 2.2 連携先（自動報告ツール）

- Supabase に `tasks` テーブル（RLS：ログインユーザー本人のみ）。
- `web/` に Next.js のタスクボード UI（ログイン・一覧・編集パネル）。
- 日次レポート・GitHub Actions 等は本リポジトリ外の責務として残す想定。

### 2.3 既存 `tasks` スキーマ（連携先・現行）

| 列 | 型・制約 | 備考 |
| --- | --- | --- |
| `id` | uuid, PK | |
| `user_id` | uuid, FK → `auth.users` | RLS の主体 |
| `title` | text | |
| `status` | text | `未着手` / `対応中` / `完了` |
| `sub_status` | text | **v1 拡張後は廃止予定**（§5.8） |
| `genre` | text | 例：`検査` / `システム` / `その他`。**維持** |
| `due_date` | date, nullable | |
| `created_at` / `updated_at` | timestamptz | |

### 3.1 UI の場面分担（O-1 確定）

| 利用場面 | 正とするクライアント |
| --- | --- |
| スマホからのタスク追加・軽い更新 | **自動報告ツール `web/`** |
| PC でのプロジェクト割当・サブタスク編集・4 ペイン作業 | **workspace-ui-kit** |

両クライアントは **同一 Supabase プロジェクト・同一認証**を前提とする。

## 4. ドメインと 4 ペイン責務

領域は **プロジェクト・タスク管理**。採用サンプルのペイン責務を次のように置き換える。

| ペイン | 採用サンプル（参考） | 本仕様での責務 |
| --- | --- | --- |
| Pane 1 | 部署 / ポジション | **プロジェクト**（ナビ軸） |
| Pane 2 | 候補者リスト | **タスク一覧**（選択中プロジェクトに紐づく） |
| Pane 3 | 候補者ダッシュボード | **タスクハブ**（1 件の中心画面） |
| Pane 4 | 選考ステージ詳細 | **サブタスク**（チェックリスト的なサブ作業面） |

### 4.1 Pane 1（プロジェクト）

- 主軸は **プロジェクト**のナビだが、**v1 の階層はフラット**（親子なしのプロジェクト一覧）。
- 採用サンプルの「部署 → ポジション」2 段ツリーは v1 では採用しない。
- **`project_id` が null のタスク**向けに、Pane 1 に **「未割当」** エントリを置く（O-7 確定）。
- **改修範囲（O-19 確定）**: `PositionPane` / `Workspace` の `Department[]` 前提をやめ、フラットな `Project[]` 向けに **Pane 1 を作り直す**。未割当エントリも同ペインに載せる。

### 4.2 Pane 3（タスクハブ）

- 日常で最も触る **1 件のタスク**を中心に置く。
- 属性編集の本拠は Pane 3 とし、Pane 4 はサブ作業に寄せる（採用サンプルの「人物ハブ + ステージ詳細」と同型）。
- **インライン編集対象（O-4 確定）**: タイトル・ステータス・期限・プロジェクト。
- **`genre`（O-8 確定）**: PC の Pane 3 でも **インライン編集**する。バッジ表示と編集 UI を同居させる。スマホ `web/` でも引き続き編集可能。

### 4.3 Pane 4（サブタスク）

- Pane 3 からドリルダウンする **1 つのサブ作業面**。
- 採用サンプル同様、未選択時の畳み・開閉の挙動を流用する想定。
- **UI（O-2 確定）**: タイトル + 完了チェックの行を並べ、**ボタンで行を増やす**チェックリスト。
- **行削除（O-10 確定）**: 行削除は **可**。確認ダイアログは **出さない**（誤操作は再追加で足す）。
- v1 では **並べ替え UI は作らない**（`sort_order` 列は先に持つ）。

## 5. データ・連携方針

### 5.1 スキーマ拡張（v1 案）

| 決定 | 内容 |
| --- | --- |
| 方針 | **Supabase を拡張**してから 4 ペインに合わせる |
| 理由 | 現行 `tasks` のみではプロジェクト軸・サブタスク行を表現できない |

#### `projects`（新規）

| 列 | 備考 |
| --- | --- |
| `id` | uuid, PK |
| `user_id` | RLS 主体（`tasks` と同様に本人のみ） |
| `name` | 表示名 |
| `sort_order` | Pane 1 の表示順 |
| `created_at` / `updated_at` | 任意（実装時に確定） |

v1 では **名前 + 表示順**のみ。色や説明は持たない。

#### `tasks`（既存拡張）

| 変更 | 備考 |
| --- | --- |
| `project_id` | uuid, nullable, FK → `projects` **`ON DELETE SET NULL`**（O-9 確定）。null は **未割当** |
| `sub_status` | **廃止**（O-3 確定）。廃止手順は §5.8 |
| `genre` | **維持** |

#### `subtasks`（新規）

| 列 | 備考 |
| --- | --- |
| `id` | uuid, PK |
| `task_id` | FK → `tasks`。`ON DELETE CASCADE`（§5.6） |
| `user_id` | RLS 用（`tasks` と整合）。INSERT 時は親 `tasks.user_id` を **アプリが引き継ぐ**（O-17 確定） |
| `title` | 必須 |
| `is_done` | boolean（完了フラグ） |
| `sort_order` | 整数。追加は **末尾**。将来のドラッグ並べ替えを見据える |
| `created_at` / `updated_at` | 任意（実装時に確定） |

v1 では **タイトル + 完了 + 並び**のみ。ステータス三値・期限・メモは持たない。

#### RLS・インデックス（O-13 確定）

`tasks` と **同型の本人 CRUD**（`auth.uid() = user_id`）を `projects` / `subtasks` にも適用する。一覧用 index は次を想定する。

| 対象 | index | 用途 |
| --- | --- | --- |
| `projects` | `(user_id, sort_order)` | Pane 1 の表示順 |
| `tasks` | `(user_id, project_id)` | Pane 2 のプロジェクト別一覧 |
| `subtasks` | `(task_id, sort_order)` | Pane 4 の並び |

### 5.2 クライアント分担

| クライアント | 役割 |
| --- | --- |
| **自動報告ツール `web/`** | **スマホ向け簡易クライアント**。タイトル・期限・ステータス・`genre` 中心の軽い追加・更新 |
| **workspace-ui-kit** | **PC 向け 4 ペイン**。プロジェクト割当・サブタスク編集など深い作業 |

### 5.3 正本データと実装フェーズ

- 業務データの正本は **Supabase**。
- `workspace-ui-kit` の JSON は、**先に UI 検証**（Pane 責務・ラベル・導線の固定）に使い、その後 Supabase 接続へ差し替える（実装順序確定）。
- 本番運用では JSON シードに依存しない。
- **通知部品（O-15 確定）**: 失敗時トースト用の部品（`sonner` 等）は **JSON フェーズ**から導入し、接続フェーズでも同じ挙動を使う。

### 5.4 認証（O-5 確定）

- **workspace-ui-kit** に、**自動報告ツール `web/` と同型**の Supabase Auth（ログイン画面・middleware・サーバー／クライアント用 Supabase クライアント）を導入する。
- 別 Next アプリのため **セッションはアプリごと**だが、同一ユーザー・同一 `tasks` を参照する。
- クロスドメインでのセッション共有は v1 の要件としない。

### 5.5 同期・保存（O-6 確定）

- **Supabase Realtime は v1 で使わない**。
- 保存は **成功時のみ UI に反映**する。失敗時は **トースト通知 + 再取得**。
- 複数端末の同時編集は **最後の書き込み優先**（明示的な競合解決 UI は v1 に含めない）。

### 5.6 削除

- **タスク削除**時、紐づく **subtasks は DB の `ON DELETE CASCADE` でまとめて削除**する。
- **確認ダイアログ（O-14 確定）**: **タスク削除**と **プロジェクト削除**のいずれも、実行前に **確認ダイアログを必須**とする。サブタスク行削除は §4.3 のとおり確認なし。

### 5.7 マイグレーション・未割当（O-7 確定）

- 既存 `tasks` 行は **`project_id = null` のまま**載せられる。
- Pane 1 の **「未割当」** に集約表示する。
- デフォルトプロジェクトの自動作成は **行わない**。
- **一覧取得（O-18 確定）**: 「未割当」選択時の Pane 2 は、DB クエリで **`project_id IS NULL`** を絞り込む。クライアント側での事後フィルタは採用しない。

### 5.8 `sub_status` 廃止手順（O-12 確定）

| 段階 | 内容 |
| --- | --- |
| 1 | 連携先 `web/` の型・フォーム・一覧更新から `sub_status` の読み書きを除去する（`web/lib/types.ts`・`TaskBoard.tsx`・`TaskEditPanel.tsx` 等） |
| 2 | 日次レポート側の `sub_status` 参照を除去する（`lib/supabase-tasks.mjs`・`lib/report.mjs` 等） |
| 3 | 上記をデプロイし、動作確認する |
| 4 | DB で `sub_status` 列を `DROP COLUMN` する |

- **デプロイ方式**: **段階デプロイ**（コード除去 → 確認 → `DROP COLUMN`）。同一デプロイは採用しない。
- **既存値**: **退避しない**（§6 の自動移行なしと整合）。

### 5.9 空状態・エラー UI（O-11 確定）

| 状況 | 扱い |
| --- | --- |
| 一覧が空（未割当ゼロ件を含む） | **各ペイン本文**に短い案内文を置く |
| 初回読み込み失敗 | **ページ全体**でワークスペースをブロックし、エラーを表示する |
| 保存失敗 | §5.5 どおり **トースト + 再取得** |

### 5.10 Server / Client 境界（O-16 確定）

- 接続フェーズでは `app/page.tsx` を **`async` + Supabase 取得**とし、初回データは **Server 側**で読み込む。
- `Workspace` は **`"use client"` のまま**とし、操作 UI とクライアント state を担う。取得結果は props で渡す。

## 6. スコープ外（v1）

- 正本 UI の単一アプリへの統合（場面分担で足りる）。
- プロジェクトの多段ツリー（親 → 子 → 孫）。
- `subtasks` のドラッグ並べ替え UI（`sort_order` 列のみ先行）。
- Supabase Realtime による他端末への即時反映。
- 明示的な編集競合解決 UI。
- 日次レポート・LINE 送信・GitHub Actions（連携先リポジトリの既存責務）。
- `sub_status` 既存値の自動移行・退避。
